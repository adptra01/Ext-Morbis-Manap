# Implementation Plan: User Role Categorization System for MORBIS Extension

## Executive Summary

Transform MORBIS Extension from a Casemix-only tool into a multi-role platform where features are filtered based on user role (Casemix, Kasir, Dokter, Farmasi, Admin). This plan outlines a **3-phase approach** with high impact and moderate effort, maintaining backward compatibility with existing users.

---

## 1. Current Architecture Analysis

### Existing Structure
```
Extension Root/
├── core.js                 # Global state & config storage (DEFAULT_CONFIG)
├── popup.html/.js          # Settings UI (320-340px width)
├── init.js                 # Content script initializer (featureModules runner)
├── background.js           # Minimal (placeholder)
└── features/
    ├── openDetail.js       # Override detail buttons (same tab)
    ├── shortcutButtons.js  # Navigation shortcuts
    ├── filterPersistence.js# Save filter state
    ├── simplifyBilling.js  # Billing table simplification
    ├── scrollButtons.js    # Scroll to top/bottom
    ├── printOptimization.js# Coming soon (hidden)
    └── batchUploadUrl.js   # Batch document upload
```

### Key Technical Constraints
- **Manifest V3**: Content scripts declared statically in `manifest.json` (no dynamic registration)
- **Storage**: `chrome.storage.sync` with `extensionConfig` key
- **Feature Loading**: `init.js` iterates `featureModules` object; checks `currentConfig.features[key].enabled`
- **Feature Isolation**: Each feature file registers itself into `featureModules` global
- **Page Matching**: Only runs on `/v2/m-klaim/*` paths (Casemix domain)
- **State Propagation**: Storage changes trigger `window.location.reload()` (hard reload)

## 2. Proposed Role System Design

### 2.1 Role Definitions
```javascript
const ROLES = {
  CASEMIX: 'casemix',    // Default for existing users, batch upload, open detail
  KASIR: 'kasir',        // Cashier section, billing-focused
  DOKTER: 'dokter',      // Medical record shortcuts, execution pages
  FARMASI: 'farmasi',    // Pharmacy-related features (future)
  ADMIN: 'admin'         // Superuser - all features + cross-role
};
```

**Notes:**
- **Admin** bypasses all role restrictions (from user clarification: "Admin can access all pages and features")
- Roles are **self-declared** (no server synchronization required)
- Single role per user (except Admin which implicitly has all) — *future multi-role array support optional*

### 2.2 Config Schema Evolution
**Current:**
```javascript
DEFAULT_CONFIG = {

```

**After Migration (Phase 1):**

## 3. Implementation Phases

### Phase 1: Foundation (High Impact, Low Effort) — **Week 1-2**

**Goal:** Introduce role-aware config without breaking existing users.

#### Tasks

**1.1 Update core.js**
- Add `ROLES` and `ROLE_METADATA` constants
- Modify `DEFAULT_CONFIG` to include `userRole: 'casemix'` and `showOnboarding: false`
- Add `allowedRoles` property to every feature in `DEFAULT_CONFIG`
- Update `loadConfig()` migration logic:
  - If `currentConfig` exists but lacks `userRole`, set `userRole = 'casemix'` (silent auto-mapping)
  - Set `showOnboarding = true` (triggers onboarding banner)
- No changes to `saveConfig()` needed

**1.2 Update popup.html**
- Insert **Role Switcher** dropdown above features list (line 402-408 section)
  - Container: `<div class="role-switcher-container">...</div>`
  - Dropdown: `<select id="roleSelector">` with options from `ROLES`
  - Info text: small caption explaining current role
- Style the dropdown to match existing UI (green accent, rounded)
- Position: directly under "Status Extension" section, before "Fitur Tersedia"

**1.3 Update popup.js**
- Add constant `ROLES` and `ROLE_METADATA` (mirror core.js or import? — keep DRY later)
- Modify `loadConfig()` to load `userRole` and `showOnboarding`
- Add `renderRoleSwitcher()` function:
  - Populate dropdown with role options
  - Set current selection based on `currentConfig.userRole`
  - Attach `change` event to persist role change instantly
- Add `saveUserRole(role)` async function
- Add `handleRoleChange(newRole)`:
  - Update `currentConfig.userRole`
  - `await saveConfig(currentConfig)`
  - `reloadPage()` (as per requirement: page refresh for role change)
- Modify `renderFeatures()` to **filter by role**:
  - For each feature, check if `currentConfig.userRole` is in `feature.allowedRoles` OR `currentConfig.userRole === 'admin'`
  - If not allowed, skip rendering (do not show toggle)
  - Adjust `totalCount` to count only allowed features
- Add **Onboarding Banner** (top of popup, below header):
  - Condition: `currentConfig.showOnboarding === true`
  - Display: green banner `"Ekstensi kini mendukung multi-role! Anda menggunakan profil: Casemix. [Ubah Role]"` with edit link
  - "Ubah Role" link opens role dropdown scroll + focuses it
  - Dismiss button (×) sets `showOnboarding = false` and saves
- Update `updateUI()` to call `renderRoleSwitcher()` and show/hide banner
- Ensure `reloadPage()` still works as expected

**1.4 Update init.js**
- Early gate: after `loadConfig()`, check `currentConfig.userRole`
- Pass role to feature loading decision:
  - `featureModules` already have `allowedRoles` defined in their registration
  - Modify loop: `if (featureConfig.enabled && isRoleAllowed(currentConfig.userRole, featureConfig.allowedRoles)) { module.run() }`
- Add `isRoleAllowed(userRole, allowedRoles)` helper:
  - Returns `true` if `userRole === 'admin'` OR `allowedRoles.includes(userRole)`
- Existing features will have `allowedRoles` defined in core.js (via DEFAULT_CONFIG)

#### Expected Behavior After Phase 1
- Existing users: automatically set to "Casemix" role, see onboarding banner
- New installs: first popup open shows onboarding banner with role selector
- Feature list dynamically filters based on selected role
- Admin bypasses all restrictions (all features visible)
- Role change triggers page reload to refresh content scripts

## 4. Migration Strategy for Existing Users

### Silent Auto-Mapping with Educational Notification
1. **First run after update**: `loadConfig()` detects missing `userRole`
2. Sets: `currentConfig.userRole = 'casemix'`
3. Sets: `currentConfig.showOnboarding = true`
4. Popup displays **Onboarding Banner** (not modal yet — less disruptive)
5. User can click "Ubah" to open role dropdown, or dismiss banner

### Why Not Forced Reassignment?
- Minimal friction; respects existing workflow
- High feasibility; no user data loss
- Trade-off: Users may not discover new roles immediately → mitigated by banner

### Why Not Contextual Detection?
- Too complex; requires page history analysis
- Low accuracy; user might juggle multiple roles
- Not worth the engineering effort at this stage

## 5. Permissions & Security

### No Server Sync Required
- Role changes are **local only** (user-declared)
- No API calls to SIMRS backend
- No authentication tied to role
- **Security consideration**: Admin role is self-selected — acceptable since extension is client-side only

### Feature Exposure
- Role filtering is **UI-level only**; determined in popup and `init.js`
- Content script modules **never load** if role disallows (resource-friendly)
- No security vulnerability: features are just UI helpers, not privileged operations

### Future Hardening (Optional)
- Encode role in extension storage with checksum? (overkill)
- Provide admin password prompt to unlock admin features? — **defer** to stakeholder request

---

## 6. Flexibility & Future-Proofing

### Plugin System Already In Place
- Feature modules are decoupled; adding new role-specific features is straightforward
- No manifest changes needed (all content scripts stay loaded, guard with `if (roleAllowed)` in `run()`)

### Capability-Based System (Optional Advanced)
- Instead of `allowedRoles: ['casemix']`, use `requires: ['can_batch_upload']`
- Roles map to capabilities: `{ casemix: ['can_batch_upload', 'can_simplify_billing'], admin: ['*'] }`
- Allows feature sharing across roles without listing every role
- **Recommendation**: **Skip for now**; simple `allowedRoles` array is sufficient for 5-10 roles

### Multi-Role Support (Future)
- If users need **multiple roles simultaneously** (e.g., Casemix + Kasir), evolve `userRole` from `string` → `string[]`
- Popup UI changes: multi-select checkboxes instead of single dropdown
- `isRoleAllowed()` checks `userRole.includes(role) || userRole.includes('admin')`
- **Impact**: Medium; affects storage schema, UI, filtering logic
- **Decision**: **Not in scope** for initial release; Admin covers dual-role use case

### Cross-Tab Role Sync Implementation

**In init.js:**
```javascript
// Listen for storage changes from other tabs
chrome.storage.onChanged.addListener((changes, namespace) => {
  if (namespace === 'sync' && changes.extensionConfig) {
    const newConfig = changes.extensionConfig.newValue;
    const oldConfig = changes.extensionConfig.oldValue;

    // Detect role change
    if (newConfig?.userRole && newConfig.userRole !== oldConfig?.userRole) {
      log(`Role changed from ${oldConfig?.userRole} to ${newConfig.userRole}, reloading tab`);
      window.location.reload();
    }
  }
});
```

**Popup:** Already calls `window.location.reload()` on all config saves (via `reloadPage()` helper). This will cascade to all open tabs.

## 9. Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Existing config migration fails | All users lose settings | Low | Migration logic uses defensive checks; never overwrites existing `userRole` |
| Role change not reflected in all tabs | Inconsistent UI | Medium | Cross-tab storage listener with reload |
| Admin self-selection abuse | Unauthorized feature access | Low (trusted environment) | No server-side impact; extension is client-side helper only |
| Feature forgetting `allowedRoles` | Feature shows to wrong role | Medium | Code review checklist; template in docs |
| Popup UI overflow with role selector | Poor UX on small screens | Low | Popup already 600px max-height; scroll OK |
| Users confused by new role system | Low adoption | Medium | Onboarding banner/modal with clear explanation |
